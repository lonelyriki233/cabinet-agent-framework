# Mandate Flow - 从任务到自动化执行

## 用户是最终授权源

用户不直接管理每个 worker。用户给：

- 目标
- 禁区
- 预算
- 自动化权限等级
- 需要汇报的粒度

## 权限等级

| 等级 | 名称 | 自动化范围 |
|---|---|---|
| L0 | Advisory Only | 只产出建议，不执行 |
| L1 | Draft Mandate | 可拆任务、生成 worker prompt，不运行 worker |
| L2 | Bounded Execution | 可运行低风险 worker，写本地文件，跑 checks |
| L3 | Night Autonomy | 可夜间连续 tick，但每轮有 stop gate |
| L4 | Release Candidate | 可准备 PR/release 材料，但不能公开发布 |

默认：L1。

用户明确说“授权中继层可以代理批准低风险任务”后，升到 L2/L3。

## 六步流转

1. Intake：领袖入口识别用户目标。
2. Council：策略参议组形成 draft mandate。
3. Relay：授权中继层检查权限等级和禁区。
4. Dispatch：分派给领域执行司。
5. Audit：独立监察司检查结果。
6. Archive：状态档案沉淀为下一轮上下文。

## 当用户赋予代理批准权时

可以直接进行自动化工作，但不是无限自治。

它能自动做：

- 技术调研。
- 拆任务。
- 分派 worker。
- 低风险实现。
- 本地检查。
- 失败后停止并报告。
- 根据重复错误提出 skill 更新。

它不能自动做：

- 改密钥。
- 生产部署。
- 删除大范围文件。
- 公共发布。
- 消耗不可控成本。
- 改变战略方向。

## 满足用户需求的关键判断

用户要求的是：AI 像政治机器一样持续运转，但仍受最高授权约束。

因此核心不是“更多 agent”，而是：

```text
授权边界 + 文书流 + 职能分工 + 监察纠偏 + 状态沉淀
```
