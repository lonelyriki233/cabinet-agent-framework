# CAF v1.0 蓝图：Agent Project Harness

创建时间：2026-06-27

## 1. 当前理解

CAF 不是一个具体业务项目，也不是某个垂直领域框架。

CAF 的主线应是：

> 一个能够支撑 agent 开发、维护、审查、迭代复杂大项目的 Agent Project Harness。

它的目标不是直接面向某个行业，而是具备承载复杂项目开发的底层能力。未来无论是前后端项目、Agent 项目、科研项目、服务业流程、工业流程、管理监督系统，都应能作为 CAF 管理下的项目形态被组织、执行和维护。

## 2. 需要避免的错误方向

- 不要把 CAF 做成某个具体项目。
- 不要把 CAF 做成空泛行业平台。
- 不要把 CAF 做成只有文档、目录、口号的空壳。
- 不要把 CAF 的子框架再套娃成“开发框架的框架”。
- 不要只堆 Python 脚本而忽略 CAF 原有结构：skills、hooks、memory、mcp、runtime、adapters、context units、dashboard、archive。
- 不要把具体项目内容写进 CAF 本体目标。

## 3. 当前 CAF 已具备什么

当前 CAF 已经有以下基础材料：

1. Task lifecycle
   - 任务创建、状态记录、阶段推进的基础结构。

2. Worker / Skill 概念
   - `skills/worker/`

3. Hooks
   - lifecycle hook 目录
   - hook event log

4. Memory

   - 项目经验沉淀文件

5. Host adapters
   - Hermes
   - Codex

   - OpenClaw

6. Runtime
   - tasks

   - archive
   - outputs

7. Protocol docs

8. Capability layer 雏形
   - memory object

   - capability audit

## 4. 当前 CAF 的主要问题

当前 CAF 的状态是：

> 材料很多，但强运行闭环还不够。

主要缺口：

1. 文档多，执行内核弱。
2. Skill 有，但没有稳定进入任务生命周期。
3. Hook 有，但没有成为强制 gate。
4. Memory 有，但缺统一召回、写入、证据追踪。
5. Worker 有概念，但缺标准 worker runtime。
6. Runtime 有目录，但还不能稳定驱动复杂项目全流程。
7. Dashboard/report 有雏形，但还不是项目指挥中心。
8. Artifact 没有统一注册和验收机制。
9. Orchestrator / Supervisor 角色不够硬。
10. 缺少从需求到实现、测试、审查、返工、归档的闭环。

## 5. v1.0 核心目标

v1.0 目标：

> 把 CAF 做成能管理复杂软件/Agent 项目开发全过程的最小可用 Agent Harness。

最小闭环：

```text
需求输入
→ 项目拆解
→ 任务生成
→ Worker 执行
→ Artifact 注册
→ Gate 审查
→ 失败返工
→ 通过归档
```

## 6. v1.0 必须补齐的六个核心模块

### 6.1 Project Harness Core

管理大项目级结构：

- project
- module
- milestone
- task
- dependency
- artifact
- decision
- risk
- acceptance criteria

目标：让 CAF 能表示一个复杂项目，而不是只表示一组零散任务。

### 6.2 Worker Runtime

定义 worker 如何工作：

- 接收 task packet
- 读取允许上下文
- 加载必要 skill
- 执行工具调用
- 产出 artifact
- 写 execution log
- 提交验收
- 失败时报告 blocker

目标：worker 不再是抽象概念，而是有标准执行协议。

### 6.3 Gate System

每个阶段有硬性验收门：

- intake gate
- design gate
- implementation gate
- test gate
- review gate
- security gate
- delivery gate
- archive gate

目标：hook 不只是记录事件，而是能阻止不合格产物进入下一阶段。

### 6.4 Context / Memory Engine

解决大项目上下文问题：

- 当前任务需要读什么
- 当前任务禁止读什么
- 如何压缩上下文
- 如何记录决策
- 如何沉淀失败经验
- 如何召回历史相似任务
- 如何区分事实、假设、决策、证据、偏好

目标：CAF 能长期维护项目，不靠一次性上下文。

### 6.5 Artifact Registry

所有产物必须登记：

- code
- docs
- tests
- design
- logs
- configs
- decisions
- review notes
- rollback points

每个 artifact 至少记录：

- path
- owner task
- type
- status
- hash 或版本标识
- validation result
- related decisions

目标：项目不是靠口头总结，而是靠可追踪产物推进。

### 6.6 Orchestrator / Supervisor

CAF 需要项目指挥层：

- 拆解需求
- 生成任务
- 分配 worker
- 检查进度
- 审核 artifact
- 触发 gate
- 识别 blocker
- 失败返工
- 更新 roadmap
- 归档经验

目标：主 agent 不再只是聊天，而是项目监督者。

## 7. v1.0 不做什么

v1.0 暂不做：

- 行业垂直应用
- 大而全 UI 平台
- 自动融资/商业化故事
- 复杂多租户系统
- 完整 SaaS
- 参数化模型训练闭环
- 真实机器人/VLA 执行

这些可以作为未来扩展，但不是 v1.0 重点。

## 9. 后续维护规则

以后 CAF 框架本体更新：

1. 先在 `update_requests/` 下记录需求。
2. 若是方向级变化，更新蓝图版本。
3. 若是执行级变化，新增 task 文件。
4. 完成后写明：
   - 改了什么
   - 为什么改
   - 影响哪些模块
   - 怎么验证
   - 遗留什么问题

## 10. 当前下一步

下一步应该不是继续讨论行业，也不是新建垂直框架。

下一步应做：

> 实现 v1.0 最小闭环：Project Harness Core + Artifact Registry + Gate System。

只有这条闭环跑通后，CAF 才算具备支持 agent 开发大项目的基础能力。
