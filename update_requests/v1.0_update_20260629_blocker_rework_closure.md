# CAF v1.0 更新记录：Blocker / Rework 自动闭环

时间：2026-06-29

## 本次更新

继续完善 CAF 功能性闭环，不扩展示示页。

本次在 Lifecycle Gate 基础上补齐 Blocker / Rework 的关闭闭环：

```text
原任务 gate fail
→ blocker 生成
→ rework task 生成
→ rework task gate pass
→ 原 blocker 自动 resolved
→ 原任务重新变为 ready
→ 失败/修复经验写入 memory case
```

## 新增能力

### 1. Blocker 最新状态视图

`runtime/project_harness/blockers.jsonl` 仍然是 append-only 事件流。

但 `list_blockers()` 现在返回每个 `blocker_id` 的最新状态，避免把 open/rework_created/resolved 三个事件当成三个 blocker。

### 2. Rework 通过后自动关闭 blocker

当任务满足：

```text
source = lifecycle_gate_rework
```

并且其 lifecycle gate pass 后，CAF 自动查找：

```text
blocker.rework_task_id == current_task_id
```

然后执行：

```text
blocker.status = resolved
original_task.status = ready
```

### 3. 手动关闭 blocker

新增 CLI：

```bash
python3 -m cabinet_framework.cli harness-resolve-blocker \
  --blocker-id <BLOCKER_ID> \
  --resolved-by-task-id <TASK_ID> \
  --resolution "manual resolution"
```

### 4. 自动记忆沉淀

Lifecycle Gate 自动写入两类 memory case：

```text
Gate failure: <task title>
Resolved blocker: <blocker id>
```

位置：

```text
runtime/memory_objects.jsonl
```

意义：

后续 Context / Memory Engine 能在类似任务里召回失败原因和修复经验。

### 5. Blocker / Rework 进入 Context Pack

Context Pack 现在包含：

```text
blocker_history
context_rules.active_blockers
```

项目级 context index 也包含：

```text
blockers
blocker_history
counts.blockers
```

## 验收标准

1. gate fail 创建 blocker + rework task。
2. blocker 有最新状态视图。
3. rework task gate pass 后自动 resolved 原 blocker。
4. 原任务从 blocked 回到 ready。
5. gate failure 写入 memory case。
6. blocker resolution 写入 memory case。
7. 手动 resolve CLI 可用。
8. 全量测试通过。
9. CAF gate 通过。
10. Showcase build 通过。

## 对真实工程流程的意义

这个更新让 CAF 更接近真实工程维护流程：

- 软件开发：测试失败 → 返工任务 → 修复通过 → bug/blocker 关闭。
- 工业 agent：安全 gate fail → 返工方案 → gate pass → 风险解除。
- 门店 agent：业务规则不通过 → 调整建议 → 审查通过 → 任务继续。
- 机器人/VLA：action/simulation fail → rework action proposal → safety pass → blocker resolved。

CAF 仍然不写死垂直业务，而是强化跨领域共通的开发维护流程。
